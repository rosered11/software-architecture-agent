## What Is a Warehouse System?

Core domains to model:

| Domain | Responsibility |
|--------|---------------|
| **Inventory** | Stock levels per SKU per location |
| **Inbound** | Receiving, putaway |
| **Outbound** | Pick, pack, ship (order fulfillment) |
| **Stock Movements** | All in/out/transfer audit trail |
| **Notifications** | Low stock alerts, shipment confirmations |
| **Reporting / ETL** | Analytics, operational dashboards |

## Kafka Event Flow

```
stock.received       → Inventory Service adjusts qty + location
order.created        → Fulfillment Worker generates pick list
pick.completed       → Fulfillment Worker triggers pack
pack.completed       → Fulfillment Worker triggers ship
shipment.dispatched  → Notification Service sends update
stock.adjusted       → Audit log + threshold check → low-stock alert
```

**Partition key rule**:
- `warehouse_id` for stock events (ordering per warehouse)
- `order_id` for fulfillment events (ordering per order)

---

## Critical Patterns Required

### 1. Outbox Pattern (stock.received, stock.adjusted)
- Stock changes + event publication must be atomic
- Without Outbox: crash between DB write and Kafka publish = ghost stock
- Implementation: `.NET` saves to `outbox_events` table in same transaction, Go relay polls and publishes

### 2. Idempotency on Stock Movements
- Receiving the same shipment twice must not double-count
- Key: `movement_id` (UUID generated at source), `UPSERT` on movement table

### 3. Saga — Order Fulfillment (multi-step, compensatable)
```
Pick → Pack → Ship
  ↓ failure at any step → compensate (release reservation, restock)
```
Orchestrated Saga via Fulfillment Worker (Go) — it owns the state machine.

### 4. Optimistic Locking on Inventory
- Concurrent pickers competing for the same SKU location
- PostgreSQL: `version` column on `inventory` row, reject stale writes

## Back-of-Envelope (Medium Warehouse Scale)

```
Orders/day:          10,000
Pick events/day:     ~30,000 (avg 3 items/order)
Stock adjustments:   ~5,000/day (receiving + corrections)
Total Kafka msgs:    ~50,000/day → ~0.6 msg/sec (low load)

Peak (flash sale):   10× → 6 msg/sec peak
                     → single Kafka partition handles 50k/sec → no bottleneck

PostgreSQL writes:
  inventory updates: 5,000/day  → ~0.06/sec → trivial
  movement log:      50,000/day → ~0.6/sec  → trivial

Connection pool (Go Fulfillment Worker):
  concurrency: 20 goroutines
  queries/order: ~4 (read reservation, write pick, write pack, write ship)
  hold_time: ~10ms each
  pool needed: 20 × 4 × 0.01 = 0.8 → pool of 10 is safe

Storage (1 year):
  movements: 50k/day × 365 = 18M rows × ~200 bytes = ~3.6GB → trivial
```

**Conclusion**: This system is not a scale problem at 10k orders/day.
The design challenge is **correctness** (consistency, idempotency), not throughput.

## Schema (PostgreSQL)

```sql
-- Inventory with optimistic locking
inventory (
  sku_id       UUID,
  location_id  UUID,
  qty          INT,
  reserved_qty INT,
  version      INT,          -- optimistic lock
  PRIMARY KEY (sku_id, location_id)
)

-- Idempotent stock movements
stock_movement (
  id           UUID PRIMARY KEY,   -- idempotency key
  sku_id       UUID,
  type         VARCHAR,            -- RECEIVE | PICK | ADJUST | TRANSFER
  qty          INT,
  reference_id UUID,              -- shipment_id / order_id / etc.
  created_at   TIMESTAMPTZ
)

-- Outbox for Kafka event relay
outbox_events (
  id           UUID PRIMARY KEY,
  aggregate_id UUID,
  event_type   VARCHAR,
  payload      JSONB,
  published_at TIMESTAMPTZ        -- NULL = unpublished
)

-- Orders
orders (
  id         UUID PRIMARY KEY,
  status     VARCHAR,
  created_at TIMESTAMPTZ
)

order_items (
  order_id    UUID,
  sku_id      UUID,
  qty         INT,
  location_id UUID,
  status      VARCHAR
)
```

**Key indexes:**
```sql
CREATE UNIQUE INDEX ON inventory (sku_id, location_id);
CREATE UNIQUE INDEX ON stock_movement (reference_id, type);  -- idempotency
CREATE INDEX ON outbox_events (published_at) WHERE published_at IS NULL;  -- relay poll
```

## Database Product Comparison — Pros & Cons

### A. PostgreSQL

**Best for**: Relational workloads requiring strong consistency, complex queries, and open-source tooling.

| | Detail |
|---|---|
| **License** | Free, open-source (PostgreSQL License) |
| **Managed options** | AWS RDS, Azure Database for PostgreSQL, Google Cloud SQL, Supabase |

**Pros:**
- `SELECT FOR UPDATE SKIP LOCKED` — makes Outbox relay and job queue patterns trivially correct with no extra infrastructure
- JSONB column type — stores flexible, queryable JSON (e.g. SKU attributes) with GIN index support
- Full ACID transactions — atomic writes across multiple tables in one TX (critical for Outbox pattern)
- Optimistic locking via `version` column — natively handles concurrent inventory updates without deadlocks
- Npgsql (.NET) and pgx (Go) — both first-class, actively maintained drivers
- Partial indexes, expression indexes — fine-grained index control reduces storage and speeds up specific queries
- Table partitioning — partition `stock_movement` by month as it grows, no application changes needed
- Strong community, 35+ years of production hardening
- No licensing cost — runs on any cloud or on-prem

**Cons:**
- VACUUM process required for dead tuple cleanup — needs monitoring (`autovacuum` tuning on high-write tables)
- No built-in read-write splitting — requires PgBouncer or application-level routing for read replica
- Tooling (pgAdmin) less polished than SSMS for .NET developers used to SQL Server
- Write amplification on UPDATE-heavy tables (MVCC model rewrites full row on each update)
- No native clustering for HA — requires Patroni, Citus, or cloud-managed HA setup

**Verdict for Warehouse**: **Best fit.** Already in stack. Outbox + optimistic locking + JSONB all work natively.

---

### B. MSSQL (SQL Server)

**Best for**: Enterprise .NET shops on Windows infrastructure, or teams with existing SQL Server licenses.

| | Detail |
|---|---|
| **License** | Paid (Developer Edition free for dev/test only) |
| **Managed options** | Azure SQL Database, AWS RDS for SQL Server |

**Pros:**
- Best-in-class EF Core integration — Microsoft owns both products, migrations and tooling are seamless
- SSMS (SQL Server Management Studio) — mature, feature-rich GUI; familiar to most .NET developers
- Always On Availability Groups — built-in HA and read replica with automatic failover
- Row-level security, dynamic data masking — enterprise compliance features out of the box
- Temporal tables — built-in bi-temporal history tracking (useful for stock audit without extra tables)
- In-memory OLTP (Hekaton) — extreme write throughput for specific hot tables
- Strong support from Microsoft for Azure-integrated deployments (Azure AD auth, Azure Monitor)

**Cons:**
- Licensing cost — Standard edition ~$900/core/year; Enterprise significantly more. No free production tier.
- No `SELECT FOR UPDATE SKIP LOCKED` — Outbox relay requires workaround (`sp_getapplock` or `UPDLOCK` hints), more complex to implement correctly
- Go driver (`go-mssqldb`) is less mature and community-maintained vs pgx
- Heavier resource footprint — minimum viable instance costs more RAM/CPU than PostgreSQL equivalent
- Vendor lock-in — cloud portability is harder; moving away from Azure SQL is painful
- Windows-centric tooling — less natural in Linux-native container environments (Kubernetes)

**Verdict for Warehouse**: **Valid but costly.** Only justified if your team is already on Azure SQL and has active licenses. The Outbox workaround alone is reason enough to prefer PostgreSQL.

---

### C. MySQL / MariaDB

**Best for**: Simple web applications, high-read workloads, teams with existing MySQL expertise.

| | Detail |
|---|---|
| **License** | MySQL: GPL / Commercial. MariaDB: GPL (BSL for some features) |
| **Managed options** | AWS RDS MySQL, Aurora MySQL, PlanetScale, Google Cloud SQL |

**Pros:**
- Extremely widely deployed — large community, abundant documentation
- Fast for simple read-heavy workloads (high QPS on indexed lookups)
- InnoDB engine supports ACID transactions and row-level locking
- `SELECT ... FOR UPDATE SKIP LOCKED` available in MySQL 8.0+ (InnoDB only)
- Aurora MySQL offers serverless scaling and near-zero replication lag
- MariaDB has Galera Cluster for multi-master HA

**Cons:**
- Foreign key enforcement off by default in some configurations (MyISAM engine has no FK support at all) — schema integrity risk in warehouse context
- JSON support significantly weaker than PostgreSQL JSONB — no GIN indexes, limited operators
- No partial indexes — must index full columns; less flexible for sparse query patterns
- EF Core MySQL driver (Pomelo.EntityFrameworkCore.MySql) is community-maintained, not official
- Go driver (`go-sql-driver/mysql`) is functional but less featured than pgx
- MariaDB and MySQL have diverged — not fully compatible; switching between them has surprises
- No equivalent of PostgreSQL table partitioning simplicity
- ENUM type is stored as integer internally — migrations are destructive (adding values requires table rebuild in older versions)

**Verdict for Warehouse**: **Not recommended.** Weaker JSON, weaker FK enforcement, and no ecosystem advantage for your .NET + Go stack. PostgreSQL is strictly better here.

---

### D. MongoDB

**Best for**: Document-heavy workloads — content management, catalogs, event stores with flexible schema.

| | Detail |
|---|---|
| **License** | SSPL (Server Side Public License) — restrictions on cloud hosting |
| **Managed options** | MongoDB Atlas (cloud-native) |

**Pros:**
- Flexible schema — no migrations needed when adding new fields to documents (useful for SKU attribute variety)
- Horizontal sharding built-in — scales writes across shards automatically
- Native JSON documents — no ORM mapping needed; document structure maps naturally to API payloads
- Atlas Search — built-in full-text search without Elasticsearch
- Change Streams — real-time event stream from DB changes (alternative to Outbox in some designs)
- Good for catalog/product data where schema varies significantly per category

**Cons:**
- No foreign keys — referential integrity must be enforced in application code; risk of orphaned records
- Multi-document transactions added in v4.0 but remain complex and less performant than RDBMS transactions — the Outbox pattern (write movement + event in one TX) is harder to get right
- Deeply relational warehouse schema (inventory ↔ location ↔ sku ↔ movement ↔ order) fights against document model — you end up either duplicating data or doing application-level joins
- No `SELECT FOR UPDATE SKIP LOCKED` equivalent — concurrent stock reservation requires custom distributed locking
- EF Core driver (`MongoDB.EntityFrameworkCore`) is immature — limited LINQ support, no migrations
- Optimistic locking requires manual `__v` field management — not as clean as PostgreSQL `version` column
- SSPL license — cannot self-host as a service for others without open-sourcing your code
- Aggregation pipeline syntax is verbose and harder to maintain than SQL for reporting queries

**Verdict for Warehouse**: **Wrong tool.** Warehouse data is fundamentally relational. MongoDB would require application-level joins, manual integrity enforcement, and a complex transaction model for the Outbox pattern. Do not use.

## ADR-002 — Database Selection: PostgreSQL

```
Context:  Warehouse system — relational data, optimistic locking,
          Outbox pattern, .NET + Go clients, Prometheus observability
Problem:  Which database engine for the primary operational store?
Scale:    ~12 QPS peak, ~32M rows/year, ~8GB — fits on a single node

Options:
  A. PostgreSQL   — already in stack, free, SELECT FOR UPDATE SKIP LOCKED
  B. MSSQL        — strong .NET support, licensing cost, no SKIP LOCKED
  C. MySQL        — weak FK enforcement, limited JSON
  D. MongoDB      — no FK, transactions complex, fights relational schema

Decision: PostgreSQL

Why:
  1. Already in stack — zero new operational overhead
  2. SELECT FOR UPDATE SKIP LOCKED — Outbox relay trivially correct
  3. Optimistic locking with version column — handles inventory contention
  4. Npgsql (.NET) + pgx (Go) — both first-class drivers
  5. JSONB — handles flexible SKU attributes without schema explosion
  6. ACID transactions — atomic stock_movement + outbox_event in one TX
  7. Free, no vendor lock-in

What you lose vs MSSQL:
  → Slightly less .NET tooling depth (SSMS vs pgAdmin)
  → No Windows-native clustering (irrelevant for microservices)

Expected Outcome:
  → Single DB engine across all warehouse services
  → Outbox relay uses SKIP LOCKED — no relay contention at 0.6 msg/sec
  → Zero licensing cost

Watch out for:
  → outbox_events table growth if relay falls behind — add cleanup job
  → Reporting queries on primary — use read replica (see below)
```

## PostgreSQL Deep Dive — Operational Notes

### 1. VACUUM Gotcha

#### How PostgreSQL Handles Updates (MVCC)

PostgreSQL uses **MVCC** (Multi-Version Concurrency Control). When you `UPDATE` or `DELETE` a row, PostgreSQL does **not** overwrite the old row. It marks the old row as "dead" and writes a new version alongside it.

```
Before UPDATE:
  [row v1: qty=10]  ← live

After UPDATE qty=9:
  [row v1: qty=10]  ← dead (marked, still on disk)
  [row v2: qty=9]   ← live
```

The dead row stays on disk and is not removed immediately.

#### The Problem: Dead Tuples Accumulate

In a Warehouse System, `inventory` rows are updated constantly — every pick, every adjustment. Over time:

```
inventory row after 1 week of picking:
  [row v1: qty=100]  ← dead
  [row v2: qty=99]   ← dead
  [row v3: qty=98]   ← dead
  ...
  [row v87: qty=13]  ← live ← the only one you care about
```

The table grows physically even though logically you only have one row per SKU+location.

| Problem | Effect |
|---------|--------|
| Table bloat | Disk usage grows far beyond actual data size |
| Index bloat | Indexes store dead row pointers — slow to scan |
| Slower queries | PostgreSQL skips past dead rows during sequential scans |
| `n_dead_tup` climbs | Visible sign of neglect in `pg_stat_user_tables` |

#### What VACUUM Does

`VACUUM` scans tables and reclaims space from dead rows, making it reusable.

```
After VACUUM:
  [row v87: qty=13]  ← live
  [free space]       ← reclaimed, reusable
```

| Command | What it does |
|---------|-------------|
| `VACUUM` | Reclaims dead tuple space for reuse within the table |
| `VACUUM FULL` | Rewrites entire table — returns space to OS, but **locks the table** |

#### autovacuum — The Automatic Safety Net

PostgreSQL runs `autovacuum` in the background. It triggers when dead tuples exceed:

```
50 + 0.2 × total_rows
```

**The gotcha**: for high-write tables like `inventory` and `outbox_events`, these defaults are too conservative. Dead tuples pile up between runs and query performance degrades before it kicks in.

#### Concrete Risk for Warehouse

```
inventory table:
  50,000 rows, updated ~10×/day = 500,000 dead tuples/day
  → autovacuum triggers at 10,050 → runs frequently ✅

outbox_events table:
  rows are INSERTed → UPDATEd (published_at) → DELETEd
  table stays small (~100 rows at any time)
  autovacuum trigger = 50 + 0.2 × 100 = 70 dead tuples
  → but churn rate is so high, tuples die faster than autovacuum runs
  → relay poll query slows down ⚠️
```

#### The Fix — Tune autovacuum Per Table

```sql
-- outbox_events: small table, very high churn
ALTER TABLE outbox_events SET (
  autovacuum_vacuum_scale_factor = 0.01,  -- trigger at 1% dead tuples
  autovacuum_vacuum_threshold = 10        -- trigger after just 10 dead tuples
);

-- inventory: medium table, frequent updates
ALTER TABLE inventory SET (
  autovacuum_vacuum_scale_factor = 0.05,  -- trigger at 5% dead tuples
  autovacuum_vacuum_threshold = 50
);
```

#### Monitor Dead Tuples (add to Prometheus/alerting)

```sql
SELECT relname,
       n_dead_tup,
       n_live_tup,
       round(n_dead_tup::numeric / nullif(n_live_tup, 0) * 100, 1) AS dead_pct,
       last_autovacuum
FROM pg_stat_user_tables
WHERE relname IN ('inventory', 'outbox_events', 'stock_movement')
ORDER BY n_dead_tup DESC;
```

Alert if `dead_pct > 20%` on any warehouse table.

### 2. SELECT FOR UPDATE SKIP LOCKED

#### The Problem — Two Workers, Same Rows

Your Outbox relay (Go worker) needs to poll unpublished events and publish them to Kafka. With one worker, this is simple. With two workers (for redundancy), you get a race condition:

```
Worker A:  SELECT * WHERE published_at IS NULL  → gets rows [1, 2, 3]
Worker B:  SELECT * WHERE published_at IS NULL  → gets rows [1, 2, 3]  ← same rows!

Worker A:  publishes event 1 to Kafka ✅
Worker B:  publishes event 1 to Kafka ← DUPLICATE
```

Result: Kafka receives the same stock event twice. Inventory count corrupted.

To fix this without a special query you would need Redis distributed locks, a `claimed_by` column with expiry logic, or a cleanup job. All extra infrastructure.

#### What `SELECT FOR UPDATE` Does

Locks the selected rows so no other transaction can touch them:

```sql
BEGIN;
SELECT * FROM outbox_events
WHERE published_at IS NULL
LIMIT 10
FOR UPDATE;
```

```
Worker A locks rows [1, 2, 3] → Worker B hits the same query → BLOCKS, waits
```

No duplicates, but Worker B is frozen doing nothing. Under load this creates a queue of stalled workers.

#### What `SKIP LOCKED` Adds

`SKIP LOCKED` tells PostgreSQL: **if a row is already locked, skip it — don't wait, just move on**.

```sql
BEGIN;
SELECT * FROM outbox_events
WHERE published_at IS NULL
LIMIT 10
FOR UPDATE SKIP LOCKED;
```

```
Worker A locks rows [1, 2, 3]  → publishes → commits

Worker B runs same query simultaneously
  → rows 1, 2, 3 are locked → SKIPPED
  → Worker B gets rows [4, 5, 6] instead → publishes → commits
```

Both workers run in parallel. Zero duplicates. Zero blocking. Zero extra infrastructure.

#### Full Outbox Relay Flow (Go)

```go
func (r *OutboxRelay) Run(ctx context.Context) error {
    tx, _ := r.db.BeginTx(ctx, nil)
    defer tx.Rollback()

    // Claim up to 10 unpublished events — skip any locked by another worker
    rows, _ := tx.QueryContext(ctx, `
        SELECT id, aggregate_id, event_type, payload
        FROM outbox_events
        WHERE published_at IS NULL
        ORDER BY created_at ASC
        LIMIT 10
        FOR UPDATE SKIP LOCKED
    `)

    var events []OutboxEvent
    for rows.Next() {
        var e OutboxEvent
        rows.Scan(&e.ID, &e.AggregateID, &e.EventType, &e.Payload)
        events = append(events, e)
    }
    rows.Close()

    if len(events) == 0 {
        tx.Rollback()
        return nil
    }

    // Publish to Kafka
    for _, e := range events {
        r.producer.Publish(e.EventType, e.Payload)
    }

    // Mark as published — atomically within same TX
    ids := extractIDs(events)
    tx.ExecContext(ctx, `
        UPDATE outbox_events
        SET published_at = NOW()
        WHERE id = ANY($1)
    `, pq.Array(ids))

    return tx.Commit()  // lock released here — other workers can now see these rows
}
```

Each event is published **exactly once**. The database enforces mutual exclusion with no application-level locking code.

#### Why MSSQL Cannot Do This Simply

MSSQL has no `SKIP LOCKED` syntax. The equivalent requires extra columns and a claim lifecycle:

```sql
-- MSSQL workaround
UPDATE TOP (10) outbox_events
SET claimed_by = @worker_id, claimed_at = GETUTCDATE()
OUTPUT inserted.*
WHERE published_at IS NULL
  AND (claimed_by IS NULL
       OR claimed_at < DATEADD(MINUTE, -5, GETUTCDATE()));  -- expiry for crashed workers
```

This requires: extra `claimed_by` + `claimed_at` columns, a timeout/expiry mechanism, a cleanup job for expired claims, and more application logic. PostgreSQL gives the same guarantee in one line.

#### Same Pattern for Job Queue (Fulfillment Worker)

```sql
-- Pick next pending order — skip any already claimed by another worker
SELECT * FROM pending_orders
WHERE status = 'PENDING'
ORDER BY created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED;
```

Two Fulfillment Workers never process the same order. No distributed lock needed.

#### One-Line Summary

> `SELECT FOR UPDATE SKIP LOCKED` lets multiple workers pull from the same table in parallel, each getting a different set of rows with zero overlap — the database enforces it. For Outbox relay and job queues this eliminates the need for Redis locks, claim columns, or any external coordination.
